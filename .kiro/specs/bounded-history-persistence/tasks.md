# 实施计划：有界历史持久化

## 概述

按依赖顺序收紧 `crates/app/src/search/history.rs` 的模型、归一化和持久化边界，并同步简化搜索结果合并排序。实现使用 Rust 及现有依赖，不新增 `proptest` 或其他测试依赖；测试采用现有框架、确定性表驱动案例和参考模型。持久化保持单进程写入，并以 Windows 本地同卷原子替换、其他环境尽可能原子为边界。

## 任务

- [ ] 1. 移除 pinned 模型、API 与搜索排序分支
  - [ ] 1.1 精简 History 模型和 API
    - 从 `history.rs` 删除 `pinned` 字段及 `pin`、`unpin`、`is_pinned`、`pinned_position`，保留 Serde 忽略未知字段以兼容旧 JSON。
    - 删除 pinned 测试，确认新序列化不含顶层 `pinned`。
    - _需求：1.1、1.2、1.3_
  - [ ] 1.2 简化搜索结果合并排序
    - 修改 `search_flow.rs`，删除 pinned/unpinned 拆分及固定位置排序，仅保留 boost、去重、分数降序和快捷键重排。
    - 不向 `window.rs` 增加业务逻辑。
    - _需求：1.1、4.1、4.2、4.3、4.4、4.5、4.6_

- [ ] 2. 实现统一 History 归一化
  - [ ] 2.1 实现 recent 去重和 20 上限
    - 从尾到头按 `action_key` 保留最后出现的完整元数据，恢复相对顺序后删除最早项，仅留最新 20 个不同键。
    - 让加载、记录、保存快照复用该规则，保持 `top_recent` 最新优先展示。
    - _需求：2.2、2.4、2.5、3.4、5.1、5.3、5.4_
  - [ ] 2.2 实现 entries 1000 上限和受保护新键
    - 按“最终 recent 成员优先、计数降序、`action_key` 升序”确定性保留最多 1000 项。
    - 满表插入新键时先保护新键，再选最多 999 个其他键；已有计数使用 `saturating_add(1)`。
    - 让 `record_full` 仅执行一次最终归一化，避免丢失新键保护状态。
    - _需求：2.1、2.3、2.5、3.1、3.2、3.3、5.2_
  - [ ]* 2.3 更新纯逻辑测试
    - 使用现有 `#[test]`、表驱动输入和确定性参考模型覆盖 999/1000/1001、新键保护、同计数排序、不同插入顺序、重复/超限 recent、Unicode 键和 `u32::MAX`。
    - 保留并扩充 boost 边界及 `record_full` 元数据测试，不新增测试依赖。
    - **设计属性 1–7；验证需求：2.1–2.5、3.1–3.4、4.1–4.6、5.2–5.4**

- [ ] 3. 实现加载、保存和固定临时文件
  - [ ] 3.1 提取基于 Path 的加载保存辅助函数
    - 实现内部 `load_from`、`save_to` 和临时路径辅助函数；公开 `load()`/`save()` 保持默认路径和 non-fatal 行为。
    - 加载入口清理同目录 `history.json.tmp`，只读取正式文件；读取/解析失败返回空归一化 History。
    - 加载成功立即归一化；保存克隆并归一化快照，不修改调用方对象。
    - _需求：2.6、5.1、5.5、6.3、6.4、6.6_
  - [ ] 3.2 实现固定临时文件写入同步
    - 先将归一化快照完整序列化到内存，再用 `create_new(true)` 创建同目录 `history.json.tmp`，完整写入、`sync_all()` 并关闭句柄。
    - 保存入口 best-effort 清理陈旧临时文件；创建、写入、同步或替换失败不得删除或截断正式文件。
    - 保持单进程写入，不增加锁、动态临时名或并发合并。
    - _需求：5.6、6.2、6.3、6.4、6.5、6.6_
  - [ ] 3.3 实现 Windows 写穿透同卷替换
    - Windows 使用已有 `windows` crate 调用 `MoveFileExW`，设置 `MOVEFILE_REPLACE_EXISTING | MOVEFILE_WRITE_THROUGH`，不设置复制回退。
    - 禁止“删除正式文件后 rename”回退；失败返回内部错误，公开保存非致命结束。
    - 非 Windows 测试构建使用同目录 `std::fs::rename`，保持尽可能原子边界。
    - _需求：6.1、6.2、6.5、6.6_

- [ ] 4. 完成兼容性与故障测试
  - [ ]* 4.1 更新旧 JSON 和序列化测试
    - 覆盖旧 `pinned` 为对象、数组、标量和 `null`；确认受支持数据不变且重新保存省略该字段。
    - 覆盖超限 entries/recent、重复 recent、缺省 recent/`is_directory`、无效 JSON 和错误字段类型。
    - **设计属性 8–9；验证需求：1.2、1.3、2.6、5.1–5.6**
  - [ ]* 4.2 添加固定路径和保存快照测试
    - 用标准库隔离测试目录验证临时名固定且同目录、成功后临时文件消失、JSON 完整可解析、保存不修改原 History。
    - 不引入 `tempfile`、`proptest` 或其他库。
    - **设计属性 9–10；验证需求：1.2、2.6、5.6、6.1、6.3、6.6**
  - [ ]* 4.3 添加 Windows 替换和失败保旧测试
    - 验证目标已存在时的 `MoveFileExW` 替换，以及加载/保存入口清理陈旧临时文件。
    - 注入临时创建或替换失败，断言旧文件字节不变、失败非致命、相关文件最多为正式文件加一个固定临时文件。
    - _需求：6.1、6.2、6.3、6.4、6.5、6.6_

- [ ] 5. 检查点：运行针对性测试
  - 使用现有 sccache 和固定 nightly 运行 `cargo +nightly-2026-06-26 test -p easysearch --release search::history`；若过滤路径不匹配，使用覆盖 History 与搜索合并回归的最窄单次命令。
  - 不绕过 `rustc-wrapper`，确保测试通过，如有问题向用户提问。
  - _需求：1.1–1.3、2.1–2.6、3.1–3.4、4.1–4.6、5.1–5.6、6.1–6.6_

- [ ] 6. 更新项目级有界持久化原则
  - 更新 `docs/ARCHITECTURE_REFACTOR.md` 的合适规范区段：持续生成的持久化数据必须有可验证容量/轮转上限及确定性保留/删除规则。
  - 声明本特性仅实施和验证 `History`/`history.json`，后续新增此类数据的规范必须定义上限及满额行为。
  - 更新 History 测试描述，移除 pinned，反映有界归一化、旧 JSON 兼容和原子提交验证。
  - _需求：7.1、7.2、7.3、7.4_

  - 停止可能锁定 `easysearch.exe` 的进程后运行 `cargo +nightly-2026-06-26 build -p easysearch --release`，并检查产物更新时间。
  - 不使用 watch 模式或绕过 rustc-wrapper；确保测试和构建通过，如有问题向用户提问。
  - _需求：1.1–1.3、2.1–2.6、3.1–3.4、4.1–4.6、5.1–5.6、6.1–6.6、7.1–7.4_

## 说明

- 标记 `*` 的测试子任务可选；实现任务不可跳过。
- 不新增测试或运行时依赖，设计属性以现有测试框架中的确定性参考模型和表驱动案例覆盖。
- 文件提交保持单进程写入假设，不设计跨进程锁或冲突合并。
- 原子性边界为 Windows 本地同卷且支持替换语义的文件系统；其他环境提供尽可能原子的替换。
